## 1. Тема и целевая аудитория

**Мессенджер MAX** — это сервис обмена сообщениями внутри экосистемы VK, позволяющий пользователям общаться в личных и групповых чатах в реальном времени.

В приложении пользователи могут:
- Отправлять текстовые сообщения
- Создавать личные и групповые чаты
- Получать push-уведомления
- Использовать реакции

### Основные функции MVP
- Регистрация и авторизация пользователя
- Список диалогов
- Отправка текстовых сообщений в реальном времени
- Групповые чаты
- Уведомления о новых сообщениях

### Целевая аудитория
По данным источников:
- Около 75 млн зарегистрированных пользователей мессенджера
- Среднесуточная аудитория (DAU) — около 45 млн пользователей

**Распределение по регионам:**

| Регион              | Доля пользователей |
|--------------------|-----------------|
| Центральный ФО (Москва) | Основной рынок и максимальная нагрузка (~40%) |
| Северо-Западный ФО (СПб) | Высокая плотность пользователей (~25%) |
| Урал, Сибирь, ДВ | Активный рост мобильной аудитории (~20%) |
| Южный и Северо-Кавказский ФО | Региональная аудитория (~15%) |

## 2. Расчёт нагрузки

### 2.1. Продуктовые метрики (Отправная точка)

За основу масштаба системы взяты данные об аудитории мессенджера MAX. Моделирование активности (количество сообщений на человека) опирается на средние показатели глобальных мессенджеров.

| Метрика | Значение | Источник / Обоснование |
| :--- | :--- | :--- |
| **MAU** (Monthly Active Users) | **75 млн** | Данные пресс-службы MAX [[1]](#ref1). |
| **DAU** (Daily Active Users) | **45 млн** | Среднесуточный охват MAX [[1]](#ref1). |
| **Sticky Factor** (DAU/MAU) | **60 %** | Отражает вовлеченность (45/75). |
| **Отправка сообщений в день (на DAU)** | **40 млрд** | Глобальная статистика WhatsApp (~100 млрд сообщений на ~2.5 млрд пользователей) [[2]](#ref2). |
| **Коэффициент чтения (Fan-out)** | **2.5** | *Допущение, обоснованное публичными данными WhatsApp:* 100 млрд доставленных / 40 млрд отправленных = 2.5. Личные чаты дают ×2 (один отправитель → один получатель). Групповые чаты со средним размером 5–10 участников поднимают среднее до 2.5 (т.е. в среднем каждое сообщение доставляется 2.5 получателям). Для MAX принимается аналогичный показатель [[2]](#ref2)[[5]](#ref5). |

---

### 2.2. Методология расчёта технических метрик

**Формулы перевода:**
*   `Средний RPS` = `(Кол-во пользователей × Действий в день) / Перевод суток в секунды`
*   `Пиковый RPS` = `Средний RPS × 2.0` (Суточная неравномерность: днем активнее, ночью нагрузка падает).
*   `Трафик (Bandwidth)` = `RPS × Средний размер запроса` (Считается в Мегабайтах в секунду, переводится в Гигабиты: `МБ/с × 8 / 1024`).

**Индустриальные константы (размеры данных):**
*   Размер текстового сообщения: **500 Байт** (Сами символы текста + JSON метаданные: ID чата, timestamp, sender_id) [[3]](#ref3).

---

### 2.3. Расчёт RPS (Запросов в секунду)

**Расшифровка цифр, используемых в формулах ниже:**
*   **45 млн** — это DAU (количество активных пользователей в день, взято из пункта 1).
*   **40** — среднее количество отправляемых сообщений одним пользователем в день.
*   **86 400** — количество секунд в одних сутках (24 часа × 60 минут × 60 секунд). Эта цифра нужна, чтобы перевести суточную активность в запросы в секунду.
*   **2.5 (Fan-out)** — внутренняя характеристика сервера: сколько получателей у одного сообщения (выведено из данных WhatsApp: 100 млрд доставленных / 40 млрд отправленных = 2.5). Fan-out влияет на нагрузку на БД и брокер сообщений, но **не порождает отдельный внешний API-метод**: онлайн-клиент получает сообщение через уже открытый WebSocket (сервер пушит фрейм сам), офлайн-клиент — через push-уведомление (APNs/FCM).
*   **21 сессия/день** — среднее количество открытий приложения одним пользователем в день (по данным Telegram [[1]](#ref1), принимается как ориентир для MAX).
*   **2 чата за сессию** — среднее количество чатов, которые пользователь открывает за одну сессию (проектное допущение).
*   **Пиковый RPS (×2)** — коэффициент неравномерности нагрузки. Ночью пользователи спят, а вечером нагрузка возрастает примерно в 2 раза от среднего значения.

| Метод API | Формула расчёта среднего RPS | Средний RPS | Пиковый RPS (×2) |
| :--- | :--- | :--- | :--- |
| **POST /messages/send** (отправка текста) | (45 000 000 × 40) / 86 400 сек | **~20 833** | **~41 666** |
| **GET /dialogs** (список диалогов при открытии приложения) | (45 000 000 × 21 сессий) / 86 400 сек | **~10 938** | **~21 875** |
| **GET /messages/history** (история чата при открытии) | (45 000 000 × 21 сессий × 2 чата) / 86 400 сек | **~21 875** | **~43 750** |
| **WebSocket connections** (concurrent, не RPS) | 45 000 000 × ~7% онлайн одновременно | **~3 150 000 соед.** | **~6 300 000 соед.** |
| **ИТОГО API Load** (без WebSocket) | Сумма RPS-методов | **~53 646** | **~107 291** |

---

### 2.4. Расчёт сетевого трафика (Bandwidth)

Вычисляем требуемую пропускную способность сетевого оборудования (Load Balancers, Edge-серверы / CDN).

| Тип трафика (Входящий + Исходящий) | Формула (Суммарный RPS × Размер) | МБ/сек | Gbit/s (Средний) | Gbit/s (Пик) |
| :--- | :--- | :--- | :--- | :--- |
| **Текстовый (JSON API)** | (20 833 + 10 938 + 21 875) × 500 Байт | ~25.6 МБ/с | **~0.20 Gbit/s** | **~0.40 Gbit/s** |

> **Вывод:** Текстовый трафик (`POST /messages/send` + `GET /dialogs` + `GET /messages/history`) генерирует высокую нагрузку на CPU серверов (парсинг JSON, маршрутизация миллионов соединений), но занимает ничтожно мало канала — **~0.40 Gbit/s** на пике. WebSocket-соединения (~3–6 млн concurrent) создают нагрузку на удержание соединений, но не генерируют значимого трафика при отсутствии активных сообщений.

### 2.5. Расчёт хранилища (Capacity Planning на 1 год)

Сколько жестких дисков потребуется для хранения истории сообщений за 1 год для 45 млн активных пользователей. В расчетах учитывается обязательная репликация данных `Factor 3` (сохранение одних и тех же данных на 3-х разных серверах для отказоустойчивости).

| Тип данных | Данных генерируется в сутки | За 1 год (Сырые данные) | Требуемое место (×3 Репликации) |
| :--- | :--- | :--- | :--- |
| **Тексты (PostgreSQL/Citus)** | 45 млн × 40 × 500 Байт = **~0.9 ТБ/сут.** | 0.9 × 365 = **~328.5 ТБ** | **~0.98 ПБ (Петабайт)** |

> **Что использовать:** Для ~1 Петабайта текстовых данных за год выбирается **PostgreSQL + Citus**: профиль нагрузки у мессенджера не только write-heavy, но и transactional/chat-centric — нужно атомарно записывать сообщение, обновлять `chats.last_message_id`, проверять членство в чате и поддерживать user-centric выборки диалогов. Такой профиль лучше ложится на распределённый PostgreSQL, чем на Cassandra.

## 3. Глобальная балансировка нагрузки

### 3.1. Функциональное разбиение по доменам

| Домен                | Функции                                   |
|---------------------|------------------------------------------|
| auth.max.ru          | Аутентификация, регистрация, сессии      |
| msg.max.ru           | Отправка/приём сообщений                  |
| search.max.ru        | Поиск сообщений и чатов                   |
| profile.max.ru       | Работа с профилем пользователя           |

> Разделение доменов позволяет изолировать процессы, гибко распределять нагрузку по типам запросов и точечно масштабировать инфраструктуру дата-центров под конкретные нужды (например, CPU для `search`). Подобный подход (Microservices/SOA) используется в архитектуре WhatsApp и Telegram [^1].

### 3.2. Расположение датацентров

| ДЦ           | Город           | Цель                                           |
|--------------|----------------|-----------------------------------------------|
| DC1          | Москва         | Основной рынок, самая высокая плотность пользователей |
| DC2          | Санкт-Петербург| Резервный регион, снижение задержки для СЗФО  |

### 3.3. Распределение запросов и расчёт профиля нагрузки

Для корректного расчёта мощностей (сайзинга) серверов и сетевых каналов отталкиваемся от целевой аудитории и профиля потребления. За основу взяты среднерыночные показатели крупных мессенджеров в РФ (таких как Telegram и VK) [^2].

#### 3.3.1. Базовый расчёт аудитории (Baseline)
* **DAU (Ежедневная аудитория):** 45 000 000 пользователей .
* **Активность:** ~150 фоновых и активных запросов в сутки от одного пользователя (отправка текста, загрузка медиа, синхронизация, онлайн-статусы) [^3].
* **Всего запросов в сутки:** 45 млн × 150 = **6 750 000 000** (6.75 млрд запросов).
* **Средний RPS (Average):** 6.75 млрд / 86 400 сек. ≈ 78 125. Для запаса прочности берём **78 125 RPS**.
* **Пиковый RPS (Peak):** С учётом ярко выраженных вечерних региональных всплесков используем стандартный телеком-коэффициент x2.5 [^4]. Пиковая нагрузка составит: 78 125 × 2.5 = **195 313 RPS**.

#### 3.3.2. Распределение нагрузки по доменам (типам запросов)
Паттерны использования мессенджера неравномерны. Это критично для балансировщиков (L7), так как сервисы требуют совершенно разных ресурсов.

**Обоснование долей.** Каждое открытие приложения порождает запросы сразу к нескольким доменам: `auth.` (валидация токена), `msg.` (список диалогов + WebSocket), `profile.` (аватары и имена собеседников), `search.` (поиск по списку чатов). Открытие конкретного чата дополнительно нагружает `profile.` (профили участников) и `msg.` (история). Исходя из этого профиля сессии, доли распределены следующим образом [^3]:

| Домен (Сервис) | Доля трафика | Средний RPS | Пиковый RPS | Специфика нагрузки на балансировщики |
| :--- | :---: | ---: | ---: | :--- |
| **msg.** | 58% | 45 313 | 113 281 | Удержание миллионов долгих TCP/WebSocket соединений, мелкие IOPS. Доминирует: каждое открытие приложения и каждый чат нагружают этот домен. |
| **profile.** | 22% | 17 188 | 42 969 | Преимущественно чтение (read-heavy). Загружается при каждом открытии чата (аватары, имена, статус online/offline участников). Значительно выше, чем кажется интуитивно. |
| **search.** | 12% | 9 375 | 23 438 | Поиск по списку диалогов происходит при каждом открытии приложения. Сложные текстовые запросы, высокая утилизация CPU (CPU-bound). |
| **auth.** | 8% | 6 250 | 15 625 | Валидация токена при каждом reconnect и запуске приложения. Редкие, но критичные операции (login, logout, refresh). |

#### 3.3.3. Распределение нагрузки по географии
Трафик маршрутизируется по ДЦ в зависимости от плотности населения РФ (по Федеральным округам) [^5]. Разница часовых поясов сглаживает единый федеральный пик, но для надёжности пики закладываются изолированно для каждого кластера.

| ДЦ (Регион) | Зона покрытия (ФО) | Доля аудитории | Средний RPS | Пиковый RPS |
| :--- | :--- | :---: | ---: | ---: |
| **DC1: Москва** | ЦФО, ПФО (Центр и Поволжье) | **40%** | 31 250 | 78 125 |
| **DC2: Санкт-Петербург** | СЗФО (Северо-Запад) | **25%** | 19 531 | 48 828 |
| **DC3: Новосибирск** | УФО, СФО, ДВФО (Урал, Сибирь) | **20%** | 15 625 | 39 063 |
| **DC4: Ростов-на-Дону** | ЮФО, СКФО (Юг и Кавказ) | **15%** | 11 719 | 29 297 |
| **ВСЕГО** | **Вся территория РФ** | **100%** | **78 125** | **195 313** |

#### 3.3.4. Архитектурное обоснование
1. **Асимметрия Европейской части (65% трафика):** Москва и Санкт-Петербург принимают на себя основной удар, что коррелирует с проникновением широкополосного интернета. За счёт прямого магистрального канала (пинг ~10-15 мс) [^6] эти ДЦ работают как Active-Active резерв друг для друга (Cross-DC Failover).
2. **Изоляция Сибири (DC3):** Наличие ДЦ в Новосибирске — архитектурная необходимость. Пинг от Владивостока до Москвы составляет ~100-120 мс (законы физики и маршрутизация), что делает real-time чат некомфортным. Новосибирский узел снижает задержку для Восточной РФ в 2.5–3 раза [^7].
3. **Локальная балансировка:** Полученные цифры RPS применяются для каждого ДЦ локально. Например, на `search.max.ru` в Москве в пике приходит 23 438 RPS (78 125 × 0.12 × 2.5). Это позволяет балансировщику внутри ДЦ направлять трафик на ограниченную группу мощных серверов, экономя бюджет.

### 3.4. Схема DNS балансировки

- Используется **GeoDNS** на федеральном уровне:
  - DNS-сервер возвращает разные A-записи (IP-адреса) в зависимости от геолокации клиента, определяемой по ASN провайдера или IP-адресу.
  - Запрос направляется на ближайший к пользователю ДЦ: московские пользователи получают IP DC1, сибирские — IP DC3 и т.д.
- **Как распределяется балансировка нагрузки:**
  - **Между ДЦ** трафик делится через GeoDNS по географии и текущему состоянию площадок: базово DC1/DC2/DC3/DC4 получают **40% / 25% / 20% / 15%** аудитории.
  - **При деградации ДЦ** его доля перераспределяется на ближайшие площадки с запасом по ёмкости. Например, при отказе DC4 южный трафик уходит в DC1 и частично в DC2; при отказе DC3 восточный трафик переводится в DC1/DC2 с ухудшением latency, но без потери доступности.
  - **Внутри ДЦ** L4-балансировщик распределяет TCP/QUIC-соединения по пулу L7-узлов через ECMP/consistent hashing, а L7 уже маршрутизирует запросы по доменам (`msg.max.ru`, `auth.max.ru`, `profile.max.ru`, `search.max.ru`) и применяет `least_conn`/`weighted round robin` в зависимости от типа сервиса.
  - **Для long-lived соединений** (`msg.max.ru`, WebSocket) используется sticky routing по connection hash, чтобы не переносить активную сессию между backend-узлами без необходимости.
- **Failover-сценарий (отказоустойчивость):**
  - Осуществляется непрерывный Health-check (проверка доступности) каждого ДЦ.
  - При аппаратном сбое или перегрузе регионального ДЦ (например, отказ DC4), GeoDNS в течение TTL (~60 сек) автоматически переключает A-записи для южного региона на ближайший резервный узел (DC1).
- **Как проверяется доступность сервисов:**
  - **L4/L7 health-check**: балансировщики проверяют `TCP connect`, `TLS handshake` и `HTTP /health` или `gRPC health-check` для каждого backend-сервиса.
  - **Функциональный health-check** для критичных сервисов (`auth`, `msg`) дополнительно проверяет зависимость от хранилищ: например, чтение тестового ключа из Redis, ping PostgreSQL/Citus coordinator и доступность брокера сообщений.
  - **Readiness** отделяется от **liveness**: если сервис жив, но не может писать в БД или брокер, он исключается из балансировки как `not ready`, но не перезапускается мгновенно.
  - Для защиты от флаппинга используются `success/failure thresholds`, а исключение узла из ротации происходит после нескольких подряд неуспешных проверок.

## 4. Локальная балансировка нагрузки

Локальная балансировка описывает схему распределения трафика **внутри каждого дата-центра** — от точки входа (Edge) до прикладных сервисов.

### 4.1. Механизм резервирования

Для пула L7-балансировщиков применяется схема **N + 1**:
- `N` — минимальное число узлов, необходимых для обслуживания пиковой нагрузки.
- `+1` — один горячий резервный узел, готовый принять трафик при отказе любого из рабочих.

Схема `N × 2` (полное дублирование) избыточна для данного масштаба и применяется только при межрегиональном Active-Active failover на уровне GeoDNS (раздел 3.4).

**Два слоя балансировки внутри ДЦ:**
1. **L4-слой** — предоставляется провайдером ДЦ как управляемый сервис (Virtual IP / Anycast). Принимает входящий TCP-поток и распределяет его по пулу L7-узлов методом ECMP (Equal-Cost Multi-Path).
2. **L7-слой** — пул NGINX-узлов. Выполняет TLS Termination, маршрутизацию по доменам (`msg.max.ru`, `auth.max.ru`, `profile.max.ru`, `search.max.ru`) и балансировку по прикладным сервисам методом Least Connections.

### 4.2. Расчёт количества L7-балансировщиков по ДЦ

**Исходные данные для расчёта (из раздела 2.3 и 3.3):**
- Суммарный пиковый RPS по всем ДЦ: **~195 313 RPS**
- Суммарный пиковый трафик через L7: **~0.40 Gbit/s** (текстовый трафик из раздела 2.4)
- Производительность одного L7-узла (24 CPU, NGINX): **10 274 HTTPS CPS** [^8] и **8.80 Gbit/s** [^9]

**Формулы:**

$$N_{ssl} = \left\lceil \frac{RPS_{peak}}{10\,274} \right\rceil, \quad N_{net} = \left\lceil \frac{BW_{peak}}{8.80} \right\rceil, \quad N = \max(N_{ssl},\, N_{net})$$

$$N_{total} = N + 1 \quad \text{(схема N+1)}$$

**Расчёт по каждому ДЦ:**

| ДЦ | Доля | Пиковый RPS | Пиковый трафик через L7 (Gbit/s) | N_ssl | N_net | N (рабочих) | N+1 (итого) |
| :--- | :---: | ---: | ---: | ---: | ---: | ---: | ---: |
| **DC1: Москва** | 40% | 78 125 | 0.17 | ⌈78 125 / 10 274⌉ = **8** | ⌈0.17 / 8.80⌉ = **1** | **8** | **9** |
| **DC2: Санкт-Петербург** | 25% | 48 828 | 0.11 | ⌈48 828 / 10 274⌉ = **5** | ⌈0.11 / 8.80⌉ = **1** | **5** | **6** |
| **DC3: Новосибирск** | 20% | 39 063 | 0.09 | ⌈39 063 / 10 274⌉ = **4** | ⌈0.09 / 8.80⌉ = **1** | **4** | **5** |
| **DC4: Ростов-на-Дону** | 15% | 29 297 | 0.06 | ⌈29 297 / 10 274⌉ = **3** | ⌈0.06 / 8.80⌉ = **1** | **3** | **4** |

### 5.3. Требования к консистентности

| Связка / таблица | Тип | Граница транзакции | Что нарушится при несоблюдении | Механизм обеспечения |
| :--- | :--- | :--- | :--- | :--- |
| `messages` + `chats.last_message_id` | **Strong** | Одна PostgreSQL-транзакция на chat-shard | Список диалогов покажет устаревшее превью после успешной отправки | `BEGIN … INSERT messages … UPDATE chats … COMMIT` |
| `chat_members` при отправке | **Strong** | SELECT FOR SHARE перед INSERT | Исключённый пользователь сможет записать сообщение в чат | Пессимистичная блокировка строки участника |
| `chat_members.unread_count` | **Strong** | Атомарный `UPDATE … SET unread_count = unread_count + 1` | Счётчик непрочитанных уйдёт в минус или пропустит инкремент | Атомарный инкремент в рамках той же транзакции fan-out |
| `user_sessions` | **Strong** | Запись и инвалидация в одном PostgreSQL-primary | Отозванная сессия будет считаться валидной до следующей репликации | Все операции с сессиями идут только через primary |
| `user_updates` после `messages` | **Eventual** | Асинхронный fan-out после COMMIT | Клиент не получит дельту мгновенно — увидит её при следующем `GET /updates` | Идемпотентный `GET /updates?since=seq_no` компенсирует задержку |
| `device_sync_state` | **Eventual** | Upsert после успешной доставки дельты | При повторном `GET /updates` клиент получит уже применённые записи повторно | Идемпотентность на стороне клиента: повторное применение безопасно |
| `user_presence` | **Eventual** | SET с TTL в Redis, без подтверждения | Статус online/offline может отставать до 60 сек | TTL = 60 сек — приемлемая задержка для индикатора присутствия |

| :--- | :--- | :--- | :--- | :--- |
| `users` | `user_id`, `phone` | Равномерное | Низкий — запись редкая, чтение распределено | Replica reads для проверки `sender_id` |
| `user_sessions` | `user_id`, `session_id` | User-centric | Средний — пользователи с 5 устройствами дают ×5 нагрузку | Кэш токенов в Redis (TTL = 5 мин) снижает нагрузку на PostgreSQL |
| `chats` | `chat_id` | Chat-centric | Высокий — популярные группы (>1000 участников) | Consistent hashing с vnodes в Citus; горячие чаты изолируются на отдельный worker |
| `chat_members` | `(chat_id, user_id)` | Chat-centric | Высокий — fan-out по большой группе = массовый UPDATE | Батчевый fan-out: обновления `unread_count` группируются и применяются пачкой |
| `messages` | `(chat_id, seq_no)` | Chat-centric | Высокий — активный чат = hot shard | Consistent hashing; keyset pagination по `seq_no` исключает full-scan |
| `user_updates` | `(user_id, update_seq_no)` | User-centric | Средний — burst при fan-out в большую группу | Шардирование по `user_id` разносит burst по разным worker-узлам |
| `device_sync_state` | `(user_id, device_id)` | User-centric | Низкий — один upsert на reconnect | Colocated с `user_updates` на том же шарде — нет cross-shard операций |
| `user_presence` | `user_id`, `device_id` | User-centric | Низкий — Redis Cluster равномерно хэширует ключи | Redis Cluster с 16 384 слотами; TTL = 60 сек автоматически чистит устаревшие ключи |

### 6.1. Физические проекции данных

Все таблицы разбиваются на два паттерна физического доступа:

| Проекция | Ключ физического доступа | Какие сценарии обслуживает | Основная идея |

### 6.2. Выбор СУБД по таблицам

| Таблица | Проекция | СУБД / хранилище | Где физически | Ключ шардирования | Резервирование | Причина выбора |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| `users` | Общий / вне hot-path | **PostgreSQL** | NVMe SSD | Не шардируется | 1 primary + 2 hot standby (streaming replication) | Глобальные UNIQUE по `phone` и `username` невозможны в шардированной БД без координатора. Strong consistency при регистрации обязательна. |
| `user_sessions` | Общий / вне hot-path | **PostgreSQL** | NVMe SSD | Не шардируется | 1 primary + 2 hot standby | Инвалидация токена должна быть немедленно видна всем репликам — только primary-reads для операций с сессиями. |
| `chats` | Chat-centric | **Citus** | NVMe SSD на worker-узлах | По `chat_id` | RF = 2 (Citus replication factor) | Метаданные чата colocated с `messages` и `chat_members` того же `chat_id` — транзакция `POST /messages/send` не выходит за пределы одного worker. |
| `chat_members` | Chat-centric | **Citus** | NVMe SSD на worker-узлах | По `chat_id` | RF = 2 | ACL-проверка и fan-out `unread_count` выполняются в разрезе `chat_id` — colocated с `chats` и `messages`. |
| `messages` | Chat-centric | **Citus** | NVMe SSD на worker-узлах | По `chat_id` | RF = 2 | Главная горячая таблица. Keyset pagination по `(chat_id, seq_no)` — все страницы истории на одном worker. |
| `user_updates` | User-centric | **Citus** | NVMe SSD на worker-узлах | По `user_id` | RF = 2 | Append-only журнал дельт. Colocated с `device_sync_state` — `GET /updates` читает курсор и дельту без cross-shard запроса. |
| `device_sync_state` | User-centric | **Citus** | NVMe SSD на worker-узлах | По `user_id` | RF = 2 | Upsert курсора и чтение `user_updates` — одна транзакция на одном worker. |
| `user_presence` | User-centric | **Redis Cluster** | RAM | По хэшу `user_id` (16 384 слота) | 1 primary + 1 replica на слот | Эфемерный статус: TTL = 60 сек, отсутствие ключа = offline. Персистентность не нужна — при рестарте Redis статус восстановится при следующем heartbeat. |

> **Почему не `dialogs_by_user` read-model?** В MAX от отдельной user-centric проекции `dialogs_by_user` сознательно отказались: при DAU 45 млн и fan-out 2.5 поддержание актуальности этой проекции потребовало бы дополнительных 52 083 RPS записи в user-shard при каждом fan-out. Вместо этого `GET /dialogs` в MAX строится через `chat_members.user_id` → `chats.last_message_id` с денормализованным `unread_count` — достаточно для MVP без scatter-gather, пока размер групп не превышает ~500 участников.
>
> **Почему PostgreSQL + Citus, а не Cassandra?** При `POST /messages/send` нужно атомарно: проверить членство (`SELECT FOR SHARE`), записать сообщение (`INSERT`), обновить `last_message_id` (`UPDATE`) — всё в одной транзакции на одном chat-shard. Cassandra не поддерживает multi-row транзакции без LWT, которые дороги. Citus сохраняет полную ACID-семантику PostgreSQL внутри одного шарда и добавляет горизонтальное масштабирование по `chat_id` и `user_id`.

### 6.4. Денормализация

Денормализация применяется для устранения дорогих JOIN-запросов на горячих путях. В MAX используется минимальный набор денормализаций — только те, без которых горячий путь требует cross-shard запроса или полного сканирования.

| Денормализованное поле / таблица | Где хранится | Что денормализовано | Что даёт | Цена поддержания |
| :--- | :--- | :--- | :--- | :--- |
| `chats.last_message_id` | `chats` | FK на последнее сообщение чата | `GET /dialogs` получает превью без JOIN с `messages` | UPDATE при каждом `POST /messages/send` — в той же транзакции |
| `chat_members.unread_count` | `chat_members` | Счётчик непрочитанных | Не нужен `COUNT(*)` по `messages` при открытии списка диалогов | Инкремент при fan-out, сброс при `POST /messages/read` |
| `chat_members.last_read_seq_no` | `chat_members` | Позиция чтения пользователя | Быстрое вычисление unread без сканирования истории | UPDATE при каждом `POST /messages/read` |
| `user_updates` (вся таблица) | Citus user-shard | Персональная дельта изменений | `GET /updates` читает только свой user-shard без scatter-gather по chat-shards | Append при каждом fan-out; TTL-очистка раз в сутки |

> **Почему в MAX нет `dialogs_by_user` read-model?** `dialogs_by_user` — отдельная user-centric проекция, дублирующая данные из `chats` + `chat_members`. В MAX от неё отказались: поддержание актуальности потребовало бы дополнительного fan-out при каждом сообщении (52 083 RPS записи), а `GET /dialogs` в MVP достаточно редкий запрос (пользователь открывает список диалогов несколько раз в день, а не при каждом сообщении). Денормализации `unread_count` и `last_message_id` достаточно для обслуживания `GET /dialogs` без scatter-gather при размере групп до ~500 участников.

### 6.5. Шардирование

| Паттерн | Таблицы | Ключ шарда | Сценарии | Colocated-гарантия | Ограничения |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Chat-centric** | `chats`, `chat_members`, `messages` | `chat_id` | `POST /messages/send`, `GET /messages/history`, проверка ACL | `chats`, `chat_members`, `messages` одного `chat_id` — всегда на одном worker | Cross-shard JOIN невозможен: нельзя в одном запросе объединить данные двух разных чатов |
| **User-centric** | `user_updates`, `device_sync_state` | `user_id` | `GET /updates`, reconnect, multi-device sync | `user_updates` и `device_sync_state` одного `user_id` — всегда на одном worker | Fan-out при отправке сообщения пишет в разные user-shards (по числу получателей) — это нормально |
| **Без шардирования** | `users`, `user_sessions` | — | Авторизация, поиск по `phone`/`username` | Глобальные UNIQUE-ограничения работают без координатора | Вертикальный предел: при >200 млн пользователей потребуется шардирование с внешним координатором |

**Митигация горячих шардов:**
- **Consistent hashing с vnodes:** Citus использует 32 768 виртуальных шардов, распределённых по worker-узлам. При добавлении нового worker перебалансируется только часть vnodes — без полного resharding.
- **Изоляция горячих чатов:** Citus позволяет вручную переместить конкретный shard на выделенный worker (`citus_move_shard_placement`). Чаты с >10 000 участников могут быть изолированы на отдельный worker.
- **Батчевый fan-out:** При отправке сообщения в большую группу записи в `user_updates` группируются в батч и применяются через `COPY` — снижает overhead на транзакции.
- **Replica reads для `GET /messages/history`:** Чтение истории направляется на реплику (RF=2), разгружая primary для записи.

### 6.6. API-маппинг на физические объекты

| API-вызов | Физические объекты | Транзакционная граница | Что происходит |
| :--- | :--- | :--- | :--- |
| `POST /messages/send` | `chat_members` → `messages` → `chats` → `user_updates` | **Две границы:** (1) Strong-транзакция на chat-shard; (2) Eventual fan-out на user-shards | **Шаг 1 (chat-shard, одна транзакция):** `SELECT FOR SHARE` на строку `chat_members` (проверка ACL + блокировка), `INSERT INTO messages`, `UPDATE chats SET last_message_id`. **Шаг 2 (после COMMIT, асинхронно):** батчевый `INSERT INTO user_updates` для каждого получателя на соответствующем user-shard; `UPDATE chat_members SET unread_count = unread_count + 1`. |
| `GET /messages/history` | `messages` (replica) | Нет транзакции | Keyset pagination: `WHERE chat_id = $1 AND seq_no < $cursor ORDER BY seq_no DESC LIMIT 50`. Направляется на реплику (RF=2). |
| `GET /dialogs` | `chat_members`, `chats` | Нет транзакции | `SELECT cm.chat_id, cm.unread_count, c.last_message_id FROM chat_members cm JOIN chats c USING (chat_id) WHERE cm.user_id = $1 ORDER BY c.updated_at DESC LIMIT 100`. Работает без scatter-gather благодаря `idx_members_user_id`. |
| `GET /updates` | `device_sync_state`, `user_updates` | **Одна транзакция на user-shard** | `SELECT last_update_seq_no FROM device_sync_state WHERE user_id=$1 AND device_id=$2 FOR UPDATE`; `SELECT * FROM user_updates WHERE user_id=$1 AND update_seq_no > $last ORDER BY update_seq_no LIMIT 200`; `UPDATE device_sync_state SET last_update_seq_no=$new, last_synced_at=now()`. Всё в одной транзакции — курсор и дельта colocated. |
| `POST /messages/read` | `chat_members` | Strong на chat-shard | `UPDATE chat_members SET last_read_seq_no=$seq, unread_count=0 WHERE chat_id=$1 AND user_id=$2`. |
| `POST /auth/login` | `users`, `user_sessions` | Strong на PostgreSQL primary | Поиск по `phone` (UNIQUE INDEX), `INSERT INTO user_sessions` с новым `push_token`. |
| `POST /auth/validate` | `user_sessions` | Нет транзакции | Lookup по `session_id` (PK) + проверка `expires_at`. Кэшируется в Redis на 5 мин. |
| `GET /presence` | `user_presence` (Redis) | Нет транзакции | `GET user:{user_id}:presence` — O(1). Отсутствие ключа = offline. |
| `POST /presence/update` | `user_presence` (Redis) | Нет транзакции | `SET user:{user_id}:presence … EX 60` — атомарный SET с TTL. |

## 7. Алгоритмы

### 7.1. Обзор ключевых алгоритмов

| Алгоритм | Где применяется | Почему важен |
| :--- | :--- | :--- |
| **Delta Synchronization** | `device_sync_state` + `user_updates` | Устройство хранит курсор `last_update_seq_no` и при каждом `GET /updates` получает только изменения после него. Без этого при reconnect пришлось бы пересобирать всё состояние пользователя заново — неприемлемо при DAU 45 млн. |
| **Ordered Merge по `update_seq_no` / `seq_no`** | Клиентское применение дельты | Пользовательские события применяются глобально по `update_seq_no`, а сообщения внутри чата — по `seq_no`. Без строгого порядка возникают дубли, пропуски и неправильный порядок сообщений в UI. |

### 7.2. Delta Synchronization

**Суть:** каждое устройство хранит свой курсор `last_update_seq_no` в `device_sync_state`. При `GET /updates` сервер не пересобирает состояние чатов, а отдаёт только те записи из `user_updates`, у которых:

```sql
update_seq_no > last_update_seq_no
```

**Что это даёт для MAX:**
- после reconnect клиент дочитывает только хвост изменений, а не всю историю;
- нагрузка на Citus user-shard пропорциональна числу пропущенных событий, а не размеру истории;
- идемпотентность: повторный `GET /updates` с тем же курсором безопасен — клиент получит те же записи.

**Как работает по шагам:**
1. Устройство знает свой последний применённый `update_seq_no`, например `120`.
2. Пока устройство было offline, появились события `121, 122, 123` в `user_updates`.
3. После reconnect клиент делает `GET /updates?device_id=X`.
4. userver-сервис читает курсор из `device_sync_state` (`FOR UPDATE`) и выбирает диапазон `121..123`.
5. Клиент применяет дельту локально.
6. После успешного применения сервер обновляет курсор до `123` в той же транзакции.

### 7.3. Ordered Merge по `update_seq_no` и `seq_no`

**Суть:** в MAX существуют два уровня порядка:
1. **Глобальный** — между всеми событиями пользователя: по `update_seq_no` из `user_updates`.
2. **Локальный** — внутри конкретного чата: по `seq_no` из `messages`.

Это необходимо, потому что сеть может доставить события не в порядке их создания, а fan-out записывает события в `user_updates` асинхронно.

**Что это даёт:**
- события пользователя применяются в детерминированном глобальном порядке;
- сообщения внутри чата всегда отображаются правильно, независимо от порядка прихода пакетов;
- клиент не зависит от порядка доставки по сети.

**Как работает по шагам:**
1. Клиент получает пачку событий из `user_updates` (отсортированных по `update_seq_no ASC`).
2. Применяет их последовательно по возрастанию `update_seq_no`.
3. Каждое событие содержит `chat_id` и `message_id` — клиент вставляет сообщение в локальную историю чата по `seq_no`.
4. Даже если событие с `seq_no=43` пришло раньше `seq_no=42`, в UI они встанут в правильном порядке.

| **userver Redis-компонент** | Доступ к Redis Cluster из C++ | Встроен в userver: поддерживает Redis Cluster, pipeline, pub/sub. Используется для `user_presence` и кэша токенов сессий. |
| **nginx** | L7-балансировка, TLS termination, маршрутизация HTTP/WebSocket | Reverse proxy перед userver-сервисами. Терминирует TLS, маршрутизирует HTTP и WebSocket, обеспечивает rate limiting на входе. |
| **NGINX Ingress Controller** | Управление nginx в Kubernetes | Декларативная конфигурация nginx через Kubernetes Ingress-ресурсы. Упрощает rolling update балансировщиков без ручного редактирования конфигов. |
| **userver Kafka-компонент** | Публикация и консьюминг событий уведомлений | Встроен в userver: поддерживает producer и consumer для Apache Kafka. `fanout-service` публикует событие в топик `notifications.push` после записи в `user_updates`; `ws-gateway` и `notification-service` консьюмят топик и доставляют push-уведомления клиентам. |

### 8.3. Базы данных и хранилища

| Технология | Область применения | Обоснование |
| :--- | :--- | :--- |
| **Apache Kafka** | Брокер уведомлений (`notifications.push`) | Decouples `fanout-service` от `ws-gateway` и `notification-service`. После записи в `user_updates` событие публикуется в Kafka — это позволяет масштабировать доставку уведомлений независимо от записи в БД, а также буферизует пики нагрузки (пиковый fan-out ×2 от среднего). Replication factor = 3, `min.insync.replicas = 2`. |
| **PostgreSQL 16** | `users`, `user_sessions` | Strong consistency, глобальные UNIQUE-ограничения по `phone` и `username`, ACID-транзакции. Шардирование не нужно — объём таблиц умеренный. |
| **Patroni** | HA-оркестрация PostgreSQL | Автоматический failover primary → standby через etcd-кворум. Управляет streaming replication и переключением ролей без ручного вмешательства. |
| **Citus** | `chats`, `chat_members`, `messages`, `user_updates`, `device_sync_state` | Расширение PostgreSQL с distributed tables. Шардирует горячие таблицы по `chat_id` и `user_id`, сохраняя colocated-гарантии и полную SQL-семантику внутри шарда. |
| **Redis Cluster** | `user_presence`, кэш токенов сессий | In-memory хранилище для эфемерных данных с TTL. Redis Cluster с 16 384 слотами равномерно распределяет ключи по узлам. |

### 8.5. Мониторинг, логи и трассировка

userver предоставляет встроенную интеграцию с Prometheus (метрики), OpenTelemetry (трассировка) и структурированным логированием — это снижает объём boilerplate по сравнению с ручной инструментацией.

**Схема сбора метрик:**
```
userver-сервисы          Prometheus           Grafana
(HTTP /metrics)  ──pull──▶  scrape  ──query──▶  Dashboard
                            │
                            └──alert──▶  Alertmanager ──▶  PagerDuty / Telegram
```

Каждый userver-сервис экспонирует эндпоинт `/metrics` в формате Prometheus text exposition. Prometheus scrape-ит все поды каждые 15 секунд через Kubernetes service discovery (`kubernetes_sd_configs`). Grafana подключается к Prometheus как datasource и строит дашборды в реальном времени.

**Ключевые метрики MAX:**

| Метрика | Источник | Что показывает |
| :--- | :--- | :--- |
| `http_server_requests_total{handler="/messages/send"}` | userver | RPS на отправку сообщений; алерт при отклонении от baseline ×2 |
| `http_server_request_duration_seconds{quantile="0.99"}` | userver | p99 latency по каждому handler; SLO: p99 < 200 мс |
| `pg_pool_connections_active` | userver PostgreSQL-компонент | Утилизация connection pool; алерт при > 80% |
| `fanout_lag_seconds` | fanout-service (custom) | Задержка между COMMIT на chat-shard и записью в `user_updates` |
| `kafka_consumer_lag` | Kafka JMX exporter | Отставание consumer-группы `ws-gateway` от топика `notifications.push` |
| `redis_connected_clients` | Redis exporter | Число активных соединений с Redis Cluster |
| `citus_shard_replication_lag` | Citus / pg_stat_replication | Отставание replica shard от primary; алерт при > 1 сек |

| Технология | Область применения | Обоснование |
| :--- | :--- | :--- |
| **Prometheus** | Сбор метрик сервисов и инфраструктуры | Pull-модель: scrape каждые 15 сек через Kubernetes service discovery. userver экспортирует метрики из коробки: RPS, latency, active connections, pool utilization, fan-out lag. Хранит TSDB локально; retention 15 дней. |
| **Grafana** | Дашборды, визуализация и алертинг | Единый интерфейс для метрик (Prometheus), логов (Loki) и трасс (Jaeger). Operational dashboards: API latency, fan-out lag, Kafka consumer lag, Citus shard health. Alertmanager интегрируется через Grafana Alerting. |
| **Loki** | Централизованный сбор структурированных логов | userver пишет структурированные JSON-логи — Loki индексирует их без полнотекстового парсинга. Хорошо интегрируется с Grafana для корреляции логов и метрик по `trace_id`. |
| **Jaeger** | Distributed tracing | userver поддерживает OpenTelemetry-трассировку. Jaeger хранит и визуализирует трассы запросов по цепочке сервисов — критично для диагностики задержек fan-out и `GET /updates`. Трассы коррелируют с логами Loki по `trace_id`. |

## 9. Обеспечение надёжности

### 9.1. Резервирование компонентов

| Компонент | Схема резервирования | Что защищает |
| :--- | :--- | :--- |
| **nginx / Ingress** | Минимум 3 экземпляра по схеме N+1, разнесённых по зонам доступности | Падение одного балансировщика не прерывает входящий трафик |
| **userver-сервисы** (API, fan-out, updates, auth) | Минимум 2 реплики каждого stateless-сервиса на разных worker-узлах | Падение одного pod не останавливает компонент; Kubernetes поднимает замену |
| **WebSocket Gateway** | Несколько реплик за L7, клиент переподключается через балансировщик | При падении одного узла теряется только часть активных соединений |
| **PostgreSQL** | Patroni + etcd (3 узла) + streaming replication: 1 primary + 2 hot standby + WAL-архивирование | Автоматический failover при падении primary; PITR для восстановления данных |
| **Citus coordinator** | Patroni + hot standby (аналогично PostgreSQL) | Координатор не является SPOF — метаданные шардов реплицируются |
| **Citus workers** | RF = 2 (replica shard на каждом worker) | Падение одного worker не делает данные недоступными |
| **Redis Cluster** | 1 primary + 1 replica на каждый слот + 3 Sentinel для failover | Автоматическое переключение при падении Redis primary |
| **Apache Kafka** | 3 брокера (RF=3, `min.insync.replicas=2`), разнесённых по зонам доступности | Потеря одного брокера не прерывает публикацию и консьюминг событий уведомлений |
| **Prometheus** | 2 независимых экземпляра с одинаковой конфигурацией | Мониторинг не является SPOF |
| **Grafana / Loki / Jaeger** | Минимум 2 экземпляра за балансировщиком | Контур наблюдаемости не влияет на serving path |

### 9.2. Сценарии graceful degradation

| Сценарий отказа | Что остаётся доступным | Что деградирует | Почему система не падает целиком |
| :--- | :--- | :--- | :--- |
| Падение одного nginx/Ingress | Весь входящий трафик продолжает обслуживаться | Снижается запас пропускной способности | Схема N+1: оставшиеся экземпляры принимают трафик |
| Падение одного userver-pod | API остаётся доступным | Кратковременный рост latency при перераспределении | Kubernetes перенаправляет запросы на живые реплики и поднимает новый pod |
| Падение standby PostgreSQL | Запись и чтение продолжаются | Уменьшается запас отказоустойчивости | Primary и оставшийся standby продолжают работу |
| Падение primary PostgreSQL | После failover (Patroni) сервис восстанавливается | Кратковременная недоступность auth/session (~10–30 сек) | Patroni автоматически промоутирует standby в primary |
| Падение одного Citus worker | Большинство чатов и пользователей доступны | Деградирует только часть шардов до переключения | RF=2: replica shard обслуживает запросы |
| Недоступность одного chat-shard | Остальные чаты работают нормально | Временно недоступна часть чатов по `chat_id` | Шардирование изолирует сбой по ключу, не валит весь кластер |
| Недоступность одного user-shard | Остальные пользователи получают `GET /updates` | Временно страдает часть пользователей по `user_id` | Сбой локализован по `user_id`, не затрагивает chat-shards |
| Отказ Redis Cluster | Отправка сообщений, история, auth работают | `user_presence` устаревает или недоступен; кэш токенов сбрасывается | `user_presence` — эфемерный контур, не на serving path сообщений |
| Отказ контура мониторинга | Пользовательский трафик обслуживается | Метрики, логи, трассы частично недоступны | Observability-контур не находится на serving path |
| Отказ Kafka | Отправка сообщений, история, `GET /updates` работают | Push-уведомления и WebSocket-push временно не доставляются | Kafka находится вне критического пути записи сообщений; клиент получит дельту при следующем `GET /updates` (polling fallback) |

## 10. Схема проекта

### 10.1. Описание архитектуры

| Пункт | Описание |
| :--- | :--- |
| **Регион** | Система развёрнута в одном основном регионе (Россия). Глобальная балансировка через GeoDNS направляет пользователей в ближайший ДЦ (раздел 3). |
| **Зоны доступности** | Внутри региона используются **3 зоны доступности**. Stateless-сервисы распределяются по зонам через `topology spread constraints`. |
| **Stateless-сервисы** | `nginx Ingress`, `auth-service`, `message-service`, `updates-service`, `dialogs-service`, `ws-gateway`, `fanout-service` — запускаются в нескольких репликах, разнесённых по зонам. Все написаны на C++ с использованием userver. |
| **Stateful-сервисы** | `PostgreSQL` (Patroni, 1 primary + 2 standby), `Citus` (coordinator + workers, RF=2), `Redis Cluster` (6 узлов), `Apache Kafka` (3 брокера, RF=3) — управляются через Kubernetes StatefulSet с persistent volumes. |
| **Путь сообщения** | Клиент → nginx → `message-service` → Citus chat-shard (транзакция) → `fanout-service` → Citus user-shards (`user_updates`) + Kafka топик `notifications.push` → `ws-gateway` / `notification-service` доставляют push клиенту. |
| **Путь синхронизации** | При reconnect: клиент → nginx → `updates-service` → Citus user-shard (`device_sync_state` + `user_updates`) → дельта возвращается клиенту. |
| **Observability** | Prometheus собирает метрики из userver-сервисов, Grafana визуализирует, Loki хранит структурированные логи, Jaeger — трассы. Контур наблюдаемости не находится на serving path. |
| **Цель отказоустойчивости** | Отказ одной зоны доступности не должен полностью остановить сервис: часть соединений переподключится, трафик уйдёт на живые реплики в других зонах. |

### 10.2. Сервисы и их роли

| Сервис | Технология | Роль |
| :--- | :--- | :--- |
| `nginx Ingress` | nginx + NGINX Ingress Controller | TLS termination, L7-маршрутизация HTTP и WebSocket, rate limiting |
| `auth-service` | userver (C++) | Регистрация, логин, валидация токенов сессий |
| `message-service` | userver (C++) | `POST /messages/send`: ACL-проверка, запись сообщения, обновление `last_message_id`, запуск fan-out |
| `fanout-service` | userver (C++) | Асинхронная запись в `user_updates` для каждого получателя, инкремент `unread_count`; публикует событие в Kafka топик `notifications.push` для доставки push-уведомлений |
| `updates-service` | userver (C++) | `GET /updates`: чтение курсора из `device_sync_state`, выдача дельты из `user_updates` |
| `dialogs-service` | userver (C++) | `GET /dialogs`: список чатов пользователя с `unread_count` и превью последнего сообщения |
| `ws-gateway` | userver (C++) | Постоянные WebSocket-соединения с клиентами; консьюмит топик `notifications.push` из Kafka и доставляет push-уведомления подключённым клиентам без polling |
| `notification-service` | userver (C++) | Консьюмит топик `notifications.push` из Kafka; отправляет мобильные push-уведомления (APNs / FCM) для offline-клиентов |
| `presence-service` | userver (C++) | `GET /presence`, `POST /presence/update`: чтение и запись `user_presence` в Redis |
